agentfootprint's memory architecture separates two concerns:

1. **`MemoryStore`** â€” WHERE memory entries are persisted (in-memory, Redis, DynamoDB, Postgres). A 10-method CRUD surface with tenant-aware `MemoryIdentity`.
2. **`MemoryPipeline`** â€” HOW memory is loaded / picked / formatted / written each turn. Two subflows (`read`, `write`) composed of explainable stages.

You attach a pipeline to an agent via `.memoryPipeline()`:

```ts
import { Agent, anthropic } from 'agentfootprint';
import { defaultPipeline, InMemoryStore } from 'agentfootprint/memory';

const pipeline = defaultPipeline({ store: new InMemoryStore() });

const agent = Agent.create({ provider: anthropic('claude-sonnet-4-5') })
  .system('You remember the user across sessions.')
  .memoryPipeline(pipeline)
  .build();

await agent.run('My name is Alice', {
  identity: { conversationId: 'alice-session-1' },
});
```

See the [Memory pipeline guide](/agentfootprint/guides/memory) for the full API.

## Shipped adapters

| Adapter | Purpose |
|---|---|
| `InMemoryStore` | Dev / tests. In-process Map-backed, tenant-aware, implements the full `MemoryStore` contract incl. batched `putMany`. |

## Roadmap

Real persistent backends (Redis, DynamoDB, Postgres) land in Phase 3 and implement the same `MemoryStore` interface. Tests written against `InMemoryStore` transfer directly to any backend.

## Writing a custom adapter

Implement the `MemoryStore` interface:

```ts
import type { MemoryStore } from 'agentfootprint/memory';

const myStore: MemoryStore = {
  async get(identity, id) { /* ... */ },
  async put(identity, entry) { /* ... */ },
  async putMany(identity, entries) { /* batched write â€” N â†’ 1 round-trip */ },
  async putIfVersion(identity, entry, expectedVersion) { /* OCC */ },
  async list(identity, options) { /* paginated */ },
  async delete(identity, id) { /* ... */ },
  async seen(identity, signature) { /* dedupe signal */ },
  async recordSignature(identity, signature) { /* ... */ },
  async feedback(identity, id, usefulness) { /* -1..1 */ },
  async getFeedback(identity, id) { /* aggregate */ },
  async forget(identity) { /* GDPR erasure */ },
};
```

<Aside type="caution" title="Tenant isolation">
Every call takes `MemoryIdentity` as the scoping argument. Adapters MUST prefix internal keys with `identityNamespace(identity)` to prevent cross-tenant access. A bug that passes the wrong identity surfaces as "no data" rather than a cross-tenant leak.
</Aside>

<Aside type="note" title="Batched writes">
`putMany` is the hot path for turn-end persistence (multiple messages per turn). Real backends should override it to pipeline the round-trip â€” Redis `MULTI`, DynamoDB `BatchWriteItem`, Postgres multi-row `INSERT`. Default fallback: sequential `put()` calls.
</Aside>
