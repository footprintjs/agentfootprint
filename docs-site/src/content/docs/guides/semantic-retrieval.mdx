import { Aside } from '@astrojs/starlight/components';

Recency-based recall (`defaultPipeline`) loses you context from many turns back: once a message scrolls out of the loadRecent window, it's gone. Semantic retrieval fixes that by **embedding every persisted message** and retrieving the top-k most similar to the current user turn, regardless of how many turns ago they were written.

```
User turn 1:   "I'm planning a trip to Japan in October"
User turns 2-4: [unrelated chatter]
User turn 5:   "what was I planning for October?"
                ↓
Semantic recall retrieves turn 1's message — even though it's
  4 turns back and would've been trimmed by sliding-window.
```

## Quick start

```ts
import { Agent, anthropic } from 'agentfootprint';
import { semanticPipeline, InMemoryStore, mockEmbedder } from 'agentfootprint/memory';

// For tests — deterministic, zero-dep
const pipeline = semanticPipeline({
  store: new InMemoryStore(),
  embedder: mockEmbedder(),
});

const agent = Agent.create({ provider: anthropic('claude-sonnet-4-5') })
  .system('You remember the user across turns.')
  .memoryPipeline(pipeline)
  .build();

await agent.run("I'm planning a trip to Japan in October.", {
  identity: { conversationId: 'alice' },
  turnNumber: 1,
});

// ... many turns later ...
await agent.run('what was I planning for October?', {
  identity: { conversationId: 'alice' },
  turnNumber: 42,
}); // → retrieves the Japan-trip message via vector similarity
```

<Aside type="caution" title="Production needs a real embedder">
`mockEmbedder()` is a deterministic character-frequency hash — fine for tests, useless for real semantic retrieval. Bring your own:

```ts
import type { Embedder } from 'agentfootprint/memory';
import OpenAI from 'openai';

const openai = new OpenAI();

const openAIEmbedder: Embedder = {
  dimensions: 1536,
  async embed({ text, signal }) {
    const res = await openai.embeddings.create(
      { model: 'text-embedding-3-small', input: text },
      { signal },
    );
    return res.data[0].embedding;
  },
  async embedBatch({ texts, signal }) {
    const res = await openai.embeddings.create(
      { model: 'text-embedding-3-small', input: texts as string[] },
      { signal },
    );
    return res.data.map((d) => d.embedding);
  },
};

const pipeline = semanticPipeline({
  store: new InMemoryStore(),
  embedder: openAIEmbedder,
  embedderId: 'openai-text-embedding-3-small',
});
```
</Aside>

## Pipeline topology

```
READ subflow:
  LoadRelevant        (embed user turn → store.search → top-k)
    → PickByBudget    (decider: skip-empty / skip-no-budget / pick)
    → FormatDefault   (inject as system message)

WRITE subflow:
  EmbedMessages       (one vector per new message via embedder)
    → WriteMessages   (persists with embeddings via store.putMany)
```

Reuses `PickByBudget` and `FormatDefault` unchanged from `defaultPipeline` — only the load and write stages differ. That's the memory-pipeline abstraction paying off: strategies compose from the same primitives.

## The `Embedder` interface

```ts
interface Embedder {
  readonly dimensions: number;
  embed(args: { text: string; signal?: AbortSignal }): Promise<number[]>;
  embedBatch?(args: { texts: readonly string[]; signal?: AbortSignal }): Promise<number[][]>;
}
```

`embedBatch` is optional but recommended — when present, `embedMessages` uses it to embed a whole turn's worth of messages in one API call instead of N round-trips.

## Store vector-search contract

Any `MemoryStore` that implements `search()` works:

```ts
interface MemoryStore {
  // ... existing methods ...
  search?<T>(
    identity: MemoryIdentity,
    query: readonly number[],
    options?: { k?: number; minScore?: number; tiers?: ...; embedderId?: string },
  ): Promise<readonly ScoredEntry<T>[]>;
}
```

| Backend | Search impl |
|---|---|
| `InMemoryStore` | O(n) linear scan with cosine — dev/tests |
| pgvector | `ORDER BY embedding <=> query LIMIT k` (roadmap) |
| Pinecone | Native `query()` API (roadmap) |
| Qdrant / Weaviate | Native vector search (roadmap) |

If you want to pass `embedderId` at retrieval time to filter out entries produced by a different model, both sides of the pipeline accept it.

## When to use which pipeline

| Situation | Pipeline |
|---|---|
| Short chat, exact-message recall (debugging, legal transcripts) | `defaultPipeline` |
| Long chat, want compression, baseline beat quality | `narrativePipeline` with heuristic extractor |
| Long chat, need high-quality beat recall | `narrativePipeline` with `llmExtractor` |
| **User asks about topics from deep history** | **`semanticPipeline`** |
| Want beats AND vector recall | Compose `extractBeats` + `embedBeats` yourself (future: `hybridPipeline`) |
| Compliance — read context only, no writes | `ephemeralPipeline` |

## Tuning

```ts
semanticPipeline({
  store,
  embedder,
  embedderId: 'openai-text-embedding-3-small',

  k: 20,                 // top-k retrieved per turn (picker narrows further)
  minScore: 0.5,         // drop weak matches (cosine threshold)
  tiers: ['hot'],        // retrieve only hot tier
  writeTier: 'hot',      // tag writes for tier-aware reads
  writeTtlMs: 30 * 24 * 60 * 60_000,

  reserveTokens: 512,    // forwarded to pickByBudget
  minimumTokens: 100,
  maxEntries: 10,        // lost-in-the-middle cap
});
```

## Custom query derivation

By default, `loadRelevant` embeds the last user message from `scope.messages`. Override for custom retrieval signals (e.g., combine user + recent tool-use context):

```ts
import { loadRelevant } from 'agentfootprint/memory';

const customLoad = loadRelevant({
  store,
  embedder,
  queryFrom: (scope) => {
    const msgs = scope.messages ?? [];
    const user = msgs.filter((m) => m.role === 'user').slice(-2);
    return user.map((m) => String(m.content)).join(' ');
  },
});
```

## Cross-model safety

Swap embedding models between deployments without retroactively polluting search. Set `embedderId` in the pipeline config — it's written onto every entry's `embeddingModel` field and used to filter retrievals:

```ts
// Old pipeline (v1) — stored entries have embeddingModel: "voyage-2"
const v1 = semanticPipeline({ store, embedder: voyage, embedderId: 'voyage-2' });

// New pipeline (v2) — searches ignore v1 entries
const v2 = semanticPipeline({
  store,
  embedder: openai,
  embedderId: 'openai-text-embedding-3-small',
});
```

Old-model entries stay in the store (you can still `store.get` them) but don't contaminate the new-model's top-k.
