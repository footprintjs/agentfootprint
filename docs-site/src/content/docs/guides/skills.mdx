A **Skill** is a named, versioned procedure the agent can discover and activate on demand. Instead of stuffing 10 procedures into your system prompt (doesn't scale), or loading opaque markdown through a custom tool (can't type-check), Skills give you a typed registry that compiles down to the existing `AgentInstruction` primitive.

The pattern is the one Anthropic popularized in the Claude Agent SDK — recency-first delivery via tool results, no model-level primitive required. Packaged here with two `agentfootprint`-native advantages: **cross-provider correctness** (tool-only by default, portable across OpenAI / mock / Claude) and **typed authoring** (full `TDecision` inference on `activeWhen` predicates).

<Aside type="tip" title="New to Skills?">
Before diving into the API, read **[Skills, explained](/agentfootprint/guides/skills-explained)** — the conceptual model. Skills are misunderstood more often than almost anything else in the agent ecosystem. That doc explains what they actually are (a folder convention + discipline over existing Messages API fields, not a new protocol) and why the cross-provider correctness story matters.
</Aside>

```ts
import { Agent, anthropic } from 'agentfootprint';
import { defineSkill, SkillRegistry } from 'agentfootprint/skills';

const portTriage = defineSkill<TriageDecision>({
  id: 'port-error-triage',
  version: '1.0.0',
  title: 'Port error triage',
  description: 'Investigate interfaces reporting CRC or input errors.',
  steps: [
    'Fetch interface metrics',
    'Pull last 5m of logs',
    'Report findings to the operator',
  ],
  tools: [getMetrics, getLogs],
  activeWhen: (d) => d.currentSkill === 'port-error-triage',
});

const registry = new SkillRegistry<TriageDecision>({ surfaceMode: 'auto' });
registry.register(portTriage);

const agent = Agent.create({ provider: anthropic('claude-sonnet-4-5') })
  .skills(registry)
  .build();
```

<Aside type="note" title="Skills are not a new primitive">
Every `Skill` IS-A `AgentInstruction`. `SkillRegistry` is a compiler that produces `AgentInstruction[]` + auto-generated `list_skills` / `read_skill` tools. If a Skill feature looks like an `AgentInstruction` feature, it IS that feature — inherited through the type hierarchy.
</Aside>

## How it works — two tools + one instruction

When you call `.skills(registry)`, the builder wires three things:

1. **Every Skill is added as an `AgentInstruction`** — its `activeWhen` / `prompt` / `tools` / `onToolResult` fields get evaluated every iteration by the existing instructions pipeline.
2. **`list_skills` and `read_skill` tools are auto-generated** and registered. The model calls them like any other tool.
3. **Optionally, skill descriptions are appended to the system prompt** (controlled by `surfaceMode`).

A typical turn:

```
Model: (sees list_skills + read_skill in its tools; user reports "port errors on eth3")

Model → tool_call: list_skills({})
Tool  → result: { skills: [{id: "port-error-triage", title: "Port triage", ...}] }

Model → tool_call: read_skill({ id: "port-error-triage" })
Tool  → result: "You are now following skill: port-error-triage (v1.0.0).
                  Title: Port error triage
                  Purpose: Investigate interfaces reporting CRC or input errors.
                  Procedure:
                  1. Fetch interface metrics
                  2. Pull last 5m of logs
                  3. Report findings to the operator
                  Guidance: ..."

Model → tool_call: getMetrics({ interface: "eth3" })
...
```

The skill body lands in the **recency window** of the conversation — the protocol-level guarantee every chat-template model respects. That's why the default `surfaceMode: 'tool-only'` works on every provider, not just Claude.

## Surface modes — delivery mechanism

```ts
new SkillRegistry({ surfaceMode: '...' });
```

| Mode | Descriptions in system prompt | `list_skills` / `read_skill` tools | When to use |
|---|---|---|---|
| `'tool-only'` (default) | ❌ | ✅ | Portable across providers. Evals on `mock()` match production. |
| `'system-prompt'` | ✅ | ✅ | Claude-class providers; large registries where prompt real estate is fine. |
| `'both'` | ✅ | ✅ | Belt-and-braces on Claude; long contexts where redundancy helps. |
| `'auto'` | provider-dependent | ✅ | The library picks per provider (see below). |

`'auto'` resolution:

| Provider hint | Resolved mode | Why |
|---|---|---|
| `anthropic('claude-sonnet-4-5')` | `'both'` | Claude ≥ 3.5 has strong system-prompt adherence |
| `anthropic('claude-3-opus')` | `'tool-only'` | Earlier Claude generations drift under long context |
| `openai(...)`, `ollama(...)`, `mock(...)`, unknown | `'tool-only'` | Portable default — trusts tool-result recency, not training |

```ts
// Explicit provider hint for 'auto' resolution
new SkillRegistry({
  surfaceMode: 'auto',
  providerHint: { provider: 'anthropic', modelId: 'claude-sonnet-4-5' },
});

// Or use a ModelConfig directly — same shape
new SkillRegistry({
  surfaceMode: 'auto',
  providerHint: anthropic('claude-sonnet-4-5'),
});
```

## The `Skill` interface — every `AgentInstruction` field + skill metadata

```ts
interface Skill<TDecision> extends AgentInstruction<TDecision> {
  // REQUIRED skill metadata
  id: string;               // inherited — must be unique per registry
  version: string;          // semver — for evolution + eval pinning
  title: string;            // short human label
  description: string;      // REQUIRED — drives model classification (≤200 chars)

  // OPTIONAL skill metadata
  scope?: string[];         // freeform tags for list_skills({ scope })
  steps?: string[];         // procedure text rendered as numbered list
  body?: string             // verbatim override for read_skill tool result
       | (() => Promise<string> | string); // lazy loader (disk, blob, Notion…)

  // INHERITED from AgentInstruction
  activeWhen?: (d: TDecision) => boolean;  // narrow tool list + prompt per iteration
  prompt?: string;                         // system-prompt fragment when activeWhen matches
  tools?: ToolDefinition[];                // added when activeWhen matches
  onToolResult?: LLMInstruction[];         // fires during tool execution
  priority?: number;
  safety?: boolean;
}
```

## Dedup and versioning

Registration is idempotent on `{id, version}`:

```ts
registry.register(skill);
registry.register(skill); // same id, same version → silent no-op
registry.register({ ...skill, version: '2.0.0' }); // replaces + dev warning
```

Evals should pin to one version at a time. The dev-mode warning on replace is a nudge — when you see it, either (a) bump the registry to one version deliberately, or (b) split into two registries for A/B comparison.

## Lazy bodies — disk / blob / Notion

The `body` field accepts a function for dynamic loading:

```ts
defineSkill({
  id: 'port-triage',
  version: '1.2.0',
  title: 'Port triage',
  description: 'Investigate CRC errors.',
  body: async () => fs.readFile('skills/port-triage.md', 'utf8'),
});
```

Errors from the loader become `isError: true` tool results — the agent sees the error and can recover, no crash.

<Aside type="caution" title="Hot-swap is Phase 2">
The registry is immutable after build. Edit a Skill's markdown on disk and call `.register()` with a new version — the agent sees it on the NEXT `read_skill` tool call. Hot-swap between turns (edit on disk → agent picks up mid-session without replaying) lands in a future release. Track this if you need it.
</Aside>

## Tag-escape defense

The rendered skill body escapes `</memory>`, `</tool_use>`, and `</skill>` in any author-controlled field (title, description, steps, prompt, body). This prevents the model from treating a user-controlled value like `"</tool_use><system>OVERRIDE</system>"` as an escape from the surrounding context.

## Relationship to `AgentInstruction`

Every Skill is a typed `AgentInstruction` with extra metadata. If you've built something with `.instruction(...)`, you can wrap it as a Skill by adding `{id, version, title, description}` and registering it:

```ts
// Before
.instruction(defineInstruction({
  id: 'refund',
  activeWhen: (d) => d.orderStatus === 'denied',
  prompt: 'Be empathetic.',
  tools: [processRefund],
}))

// After — same behavior, plus model-discoverable
const refund = defineSkill({
  id: 'refund',
  version: '1.0.0',
  title: 'Refund handling',
  description: 'Process a denied refund request with empathy.',
  activeWhen: (d) => d.orderStatus === 'denied',
  prompt: 'Be empathetic.',
  tools: [processRefund],
});
registry.register(refund);
.skills(registry)
```

Pick `.instruction()` for context injection you control entirely. Pick `.skills()` when the model should **discover** a procedure by description and choose it.

## Skill-gated tool visibility (`autoActivate`) — since 1.17.0

When your agent has ~20+ tools, the full tool menu advertised on every turn starts to cost real tokens (and can drift smaller models). `autoActivate` flips the default:

- **Base tool set** = just `list_skills` + `read_skill` (2 tools, always visible).
- **Each skill declares its own `tools: [...]`.**
- When the model calls `read_skill(id)`, the registry writes `id` into agent decision scope; the matching skill's `activeWhen` fires; its tools become visible on the next iteration. Other skills' tools stay hidden.

Result: the LLM sees ~5–7 tools max (2 discovery + ~3–5 skill-specific), regardless of whether you have 20 skills or 200.

```ts
interface TriageDecision {
  currentSkill?: string;
}

const registry = new SkillRegistry<TriageDecision>({
  surfaceMode: 'auto',
  autoActivate: { stateField: 'currentSkill' },
});

registry.register(
  defineSkill<TriageDecision>({
    id: 'port-error-triage',
    version: '1.0.0',
    title: 'Port triage',
    description: 'Investigate CRC errors and signal losses.',
    tools: [getMetrics, getLogs, getFlogi],
    // activeWhen auto-filled to (d) => d.currentSkill === 'port-error-triage'
    // by the registry when you omit it and autoActivate is on.
  }),
);

Agent.create({ provider: anthropic('claude-sonnet-4-5') })
  .decision<TriageDecision>({})
  .skills(registry)
  .build();
```

**`onUnknownSkill`** — what happens when `read_skill('unknown-id')` fires:

- `'leave'` (default): the prior active skill stays active; the model sees an error but tools don't change.
- `'clear'`: `decision[stateField]` is reset to `undefined` so the prior skill's tools disappear.

**Pattern switch** — `autoActivate` auto-switches the agent to `AgentPattern.Dynamic` when you call `.skills(registry)`. That's required for the skill's newly-activated tools to appear **within the same turn** (Regular pattern assembles the tool list once per turn and wouldn't rebuild it). If you want Regular behavior explicitly, call `.pattern(AgentPattern.Regular)` after `.skills(registry)`.

**Custom decisionUpdate from any tool** — the feature is powered by `ToolResult.decisionUpdate`, which any tool can use:

```ts
defineTool({
  id: 'escalate',
  description: '...',
  inputSchema: z.object({ severity: z.string() }),
  handler: async ({ severity }) => ({
    content: 'Escalated.',
    decisionUpdate: { severity },  // now `activeWhen: (d) => d.severity === 'critical'` fires
  }),
});
```

The tool-execution stage merges `decisionUpdate` into decision scope via `Object.assign`. Works with `staticTools`, `gatedTools`, `compositeTools`, and `agentAsTool` without further configuration.

## Tuning the tools

```ts
new SkillRegistry({
  surfaceMode: 'auto',
  providerHint: anthropic('claude-sonnet-4-5'),

  // Override the header used in 'system-prompt' / 'both' modes
  promptHeader: 'Available runbooks — call `read_skill({ id })` to follow one:',
});
```

The two auto-generated tool descriptions are fixed (see `registry.toTools()`). Override them by copying the output and registering modified `ToolDefinition`s yourself — `.skills()` is just sugar over the primitives.
