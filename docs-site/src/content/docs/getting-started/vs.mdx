> Three competitors, three sentences each, then the depth below the fold. The right framework for your team depends on what you're optimizing for; this page is meant to make that decision easier, not harder.

## In one sentence per competitor

- **vs LangChain** — agentfootprint has a 2-primitive + 4-composition substrate; LangChain ships a class-per-paper. If you're maintaining a "pile of LangChain abstractions" codebase and the abstractions feel like the problem, switch.
- **vs LangGraph** — both treat agents as graphs; agentfootprint records every traversal as a typed event for free (because the framework owns the loop, not you). If you've found yourself instrumenting LangGraph manually, switch.
- **vs CrewAI** — agentfootprint has no `Crew` / `Agent` / `Task` separate primitives; multi-agent IS composition of single agents through Sequence/Parallel/Conditional/Loop. If CrewAI's role / goal / backstory framing feels like ceremony, switch.

## Strengths agentfootprint claims

| Claim | Where it shows |
|---|---|
| Every code block in our docs is a real, runnable file in CI | The Phase 1 CodeFile + region marker infrastructure (this site) |
| The trace is a cache of the agent's thinking | [Causal memory](/agentfootprint/guides/memory/#causal-memory--replay-decisions-not-just-messages) |
| Every named pattern is a recipe, not a class | [Patterns](/agentfootprint/getting-started/key-concepts/#patterns--named-compositions) |
| Mocks-first dev → real-infra prod is one-line swap per boundary | [Quick Start](/agentfootprint/getting-started/quick-start/) |
| Multi-tenant isolation enforced at the storage boundary | [Memory guide](/agentfootprint/guides/memory/) |

## Strengths agentfootprint does NOT claim

- We are not the only framework with multi-agent. CrewAI / LangGraph have it. Ours has a smaller surface; pick on taste.
- We are not the only framework with memory. Most have some flavor. Ours is the only one with **Causal** type (decision-evidence persistence) — see [Memory guide](/agentfootprint/guides/memory/#the-four-types).
- We are not the only framework with observability. Most ship spans / events. Ours is **typed** + **emitted during DFS traversal** so it's structured + complete by construction, not collected after the fact.

## When NOT to use agentfootprint

- You need a vector-search-capable production memory store TODAY. v2.3 ships InMemoryStore + RedisStore (no vector) + AgentCoreStore (no vector). v2.6 lands pgvector / Pinecone / Qdrant adapters.
- You need full multi-modal (images, video) in messages. Our `LLMMessage.content` is `string` today.
- You're already happy with your current framework's abstractions and your team is shipping. Don't switch on hype.

## Migration from other frameworks

- **From LangChain (v0.0.x → v0.3+ era)** — most LangChain agent code maps to `Agent.create({ provider }).tool(...).build()`. Tools are `defineTool({ name, description, inputSchema, execute })`. Memory becomes `defineMemory({ type, strategy, store })`. Migration path tracked in `MIGRATION.md` (in repo).
- **From LangGraph** — `StateGraph` nodes → `Sequence` steps; conditional edges → `Conditional`; the rest of the wiring evaporates. Compositions are typed; no `START` / `END` constants needed.
- **From CrewAI** — `Crew` of agents with tasks → `Sequence(researcher, writer, editor)` etc. Roles + goals → system prompts. Tools work the same way.

## Honest comparison table

(Limited to 5 rows per coworker-review feedback — fewer is more readable.)

| Concern | LangChain | LangGraph | CrewAI | agentfootprint |
|---|---|---|---|---|
| Surface area | Large; class-per-paper | Mid; graph-DSL | Mid; role-based | Small; 2 primitives + 4 compositions |
| Observability | Manual instrumentation | Spans (LangSmith) | Spans (own service) | Typed events emitted during traversal |
| Cross-run "why?" replay | Not built in | Not built in | Not built in | **Causal memory** (decision-evidence persistence) |
| Test story | Mocks possible but not first-class | Mocks possible but not first-class | Limited | Mocks-first by design — `mock({replies})`, `mockMcpClient`, `InMemoryStore` |
| Multi-tenant isolation | Caller responsibility | Caller responsibility | Caller responsibility | Enforced at every store boundary via `MemoryIdentity` |
